# 테스트 유형 정리 (Unit / Integration / System)

본 문서는 **단위 테스트(Unit Test)**, **통합 테스트(Integration Test)**,  
**시스템 테스트(System Test / E2E)** 의 개념과 차이를 명확히 정리한다.

---

## 1️⃣ 단위 테스트 (Unit Test)

### 정의

- 하나의 **함수 또는 클래스 단위**를 테스트
- 모든 외부 의존성(DB, API, Repository 등)은 **Mock 처리**

### 목적

- 비즈니스 로직의 정확성 검증
- 리팩토링 시 안정성 확보

### 특징

- DB 연결 ❌
- HTTP 요청 ❌
- 실행 속도 **매우 빠름**
- 실패 원인 추적이 명확함

### 적용 예

- 계산 로직
- Service 내부 메서드
- Util 함수

---

## 2️⃣ 통합 테스트 (Integration Test)

### 정의

- 여러 컴포넌트(Service, Repository 등)가 **함께 동작하는지** 테스트
- 실제 DB 또는 테스트 전용 DB 사용

### 목적

- 모듈 간 연동 검증
- Entity 매핑, 쿼리, 설정 오류 조기 발견

### 특징

- DB 연결 ⭕ (테스트 DB)
- HTTP 요청 ❌ (일반적으로)
- 실행 속도 **보통**
- 실패 원인 비교적 명확

### 적용 예

- Service ↔ Repository 연동 테스트
- DB CRUD 검증
- 트랜잭션 처리 검증

---

## 3️⃣ 시스템 테스트 (System Test / E2E)

### 정의

- 실제 서비스와 거의 동일한 환경에서 **전체 흐름을 검증**
- 사용자 행동 시나리오 기반 테스트

### 목적

- “사용자가 이 기능을 실제로 사용할 수 있는가?” 검증
- 배포 전 최종 품질 확인

### 특징

- DB 연결 ⭕
- HTTP 요청 ⭕
- 인증 / 인가 포함
- 실행 속도 **느림**
- 실패 원인 추적이 어려움

### 적용 예

- 로그인 → 인증 → API 호출 전체 흐름
- 핵심 비즈니스 시나리오
- 주요 API 정상 동작 여부

| ------------- | ------------- | ------------ | ------------- |
| 테스트 범위   | 함수 / 클래스 | 모듈 간 연동 | 전체 시스템   |

| HTTP 요청     | ❌            | ❌           | ⭕            |

| 유지보수 비용 | 낮음          | 중간         | 높음          |
